# mcpmenago

**MCP Manager** — a standalone CLI tool + central MCP server that manages source code indexes for any GitHub repository. Point it at a repo, it clones and indexes it, then exposes the symbols via a single MCP server. Built on FastMCP.

Abbreviated from "MCP Manager" — easy to remember, easy to type.

## Quick Start

```bash
# Install dependencies (managed by uv)
uv sync --project ./mcpmenago --extra dev

# Add a repository
mcpmenago add https://github.com/rdkit/rdkit --lang python --lang cpp --head-ref Release_2025_09_6

# Learn which symbols your project uses (updates weights)
mcpmenago learn rdkit

# List managed books
mcpmenago list

# Show details for a book
mcpmenago show rdkit
```

## Analyze Index Script
```bash
uv run --project ./mcpmenago python scripts/analyze_index.py rdkit
```

## CLI Commands

| Command | Purpose |
|---------|---------|
| `mcpmenago add <github-url> --lang <lang> [--lang <lang>] [--head-ref <tag>] [--book-name <name>]` | Clone repo, build index |
| `mcpmenago remove <book>` | Remove book and all its data from library |
| `mcpmenago list` | List all managed books |
| `mcpmenago show <book>` | Show book config, index stats, weight summary |
| `mcpmenago rebuild <book>` | Rebuild index for a book. Weights preserved. |
| `mcpmenago rebuild --all` | Rebuild all indexes |
| `mcpmenago update <book> --head-ref <tag\|branch>` | Update to a new git ref (clean re-clone + rebuild) |
| `mcpmenago learn <book>` | Scan project imports, update weights |
| `mcpmenago uninstall` | Remove everything — library, config, gitignore section |

### `add` flow
1. Validate GitHub URL format
2. `git clone --depth 1 [--branch <head-ref>] <url>` into `mcpmenago/library/<book>/repo/`
3. Index source files → `01_index.json`
4. Write `book.json` (BookMeta)
5. Append `mcpmenago/library/` to `.gitignore` under managed section (idempotent)

### `update` flow (clean state management)
1. Delete `mcpmenago/library/<book>/repo/` entirely
2. Fresh shallow clone at new ref — no incremental git, no submodules
3. Rebuild index from scratch
4. Update `book.json` with new `head_ref` + `head_ref_resolved` (commit SHA)
5. `weights.json` is preserved — it's independent of the index

### `remove` flow
1. Delete `mcpmenago/library/<book>/` directory entirely
2. If library is now empty, clean up `.gitignore` managed section

### `learn` flow
1. Read `learn_dirs` from `mcpmenago/mcpmenago.json` (defaults: `["src", "tests", "notebooks"]`)
2. Scan all `.py` and `.ipynb` files in those directories for imports matching the book's package
3. Write `weights.json`: `{symbol_name: 0.8}` for every discovered symbol
4. Symbols not found in project imports remain at default weight (`NOT_USED = 0.3`)

### `uninstall` flow
1. Delete entire `mcpmenago/library/` directory
2. Clean `.gitignore` managed section

### Book name derivation
Extracted from GitHub URL: `https://github.com/rdkit/rdkit` -> `rdkit` (last path segment). Override with `--book-name` flag.

│   ├── models.py          # Pydantic schemas + weight constants
│   └── version.py
├── tests/                 # Test suite
├── mcpmenago.json         # Central config (user-editable, minimal)
├── pyproject.toml
├── README.md
└── library/               # .gitignored — all cloned repos + indexes
    └── <book>/            # One directory per managed repository
        ├── repo/          # Shallow clone from GitHub

        └── weights.json           # Learned weights (independent of index)

### `.gitignore` management
The CLI automatically appends a managed section to the project's `.gitignore`:
```gitignore
# [mcpmenago] automatically generated content — do not edit
mcpmenago/library/
```
All cloned repos and indexes are ignored. The tool code itself is committed.

## Central Config (`mcpmenago.json`)

Minimal, user-editable. Single source of truth for global settings.

```json
{
  "learn_dirs": ["src", "tests", "notebooks"],
  "supported_languages": ["python", "cpp"]
}
```

| Field | Purpose | Default |
|-------|---------|---------|
| `learn_dirs` | Directories to scan when running `mcpmenago learn`. Relative to project root. | `["src", "tests", "notebooks"]` |
| `supported_languages` | Languages to index. Drives `--lang` validation. | `["python", "cpp"]` |

## MCP Server Tools

Central FastMCP server (`mcpmenago/server.py`) exposes these tools. Every tool takes an explicit `package` parameter to route to the right book. STDIO transport.

| Tool | Purpose |
|------|---------|
| `get_symbol(name, package, context_file?)` | Exact symbol lookup — returns function/class body |
| `search_code(query, package, source?, context_file?)` | Tiered ripgrep search |
| `read_file(path, package)` | Read one source file |
| `inspect_module(module_name, package)` | C++ signatures from compiled .so via inspect |
| `get_module_index(package, min_weight?)` | Module map with weights |
| `list_directory(path, package)` | Browse source tree, names only |

### `.mcp.json` integration

```json
{
  "mcpServers": {
    "mcpmenago": {
      "command": "uv",
      "args": ["run", "--project", "./mcpmenago", "python", "-m", "mcpmenago.server"]
    }
  }
}
```

## Search Algorithm

The search algorithm begins when Claude makes an MCP tool call. Claude autonomously selects which tool to call based on the tool docstrings — this is the **Claude layer (L0)** sitting above all server logic.

### Tool Selection (L0 — Claude's decision)

Claude reads the tool descriptions at session start and picks one per query. The docstring IS the interface contract with Claude — not the code.

**Two primary tools:**

| Tool | When Claude calls it |
|------|---------------------|
| `get_symbol(name, package, context_file?)` | Knows the exact symbol name |
| `search_code(query, package, context_file?)` | Needs to find by text pattern |

The other four tools (`read_file`, `inspect_module`, `get_module_index`, `list_directory`) are exploration/fallback tools.

### Weights and context filtering

Independent of indexing, two mechanisms narrow and rank results at query time:

- **Static weights** (`weights.json`) — built by `mcpmenago learn`. Symbols found in project imports get `DISCOVERED = 0.8`; others default to `NOT_USED = 0.3`. Persists across index rebuilds.
- **Query context** (`context_file` param) — at call time, the server parses the file being edited for import statements and filters results to matching modules. No caching — fresh parse per request (~1ms).

### Known gaps

| Gap | Tool affected | Impact |
|-----|--------------|--------|
| `get_symbol` has no result cap | `get_symbol` | `__init__` returns 184 entries — potentially 15,000+ tokens per call |
| `source` param (`python`/`cpp`/`all`) is declared but unused | `search_code` | Claude thinks it can filter by language — it cannot |
| `modules` list is always empty | `get_module_index` | Tool always returns `{}` — effectively dead |
| L0 (tool selection) is entirely implicit | all | No visibility into why Claude chose a particular tool |

## Build Pipeline

The index is produced by a staged build pipeline. Each stage has its own output file on disk — the file naming IS the contract.

```
repo/  →  [indexer]  →  01_index.json  →  (future: 02_index.json, ...)
```

| Stage | File | Content |
|-------|------|---------|
| 01 | `01_index.json` | Raw symbol dump from indexer. No filtering. |
| 02+ | `02_*.json` | Future refinement stages (deduplication, enrichment). Added when clearly needed. |

**Why numbered files:** Intermediate artifacts are inspectable on disk. `01_index_analysis.json` (from `analyze_index.py`) measures Stage 01 quality without reading the full file.

**Separate lifecycle:** `weights.json` is NOT part of this pipeline — different trigger (`mcpmenago learn`), different input (project imports), persists across pipeline runs.

## Design Principles

- **Flat is better than nested** — minimal directory depth, simple organization
- **Simple architecture is better than complex** — one server, one config, one index per book
- **Standalone and portable** — fully contained in `mcpmenago/`, no coupling to any specific project
- **User-centered UX** — minimal effort, good error messages, foolproof defaults
- **Pydantic everywhere** — schemas, config, serialization. MongoDB-ready by design.

## Decisions Log

Key architectural decisions made during design, with rationale.

| # | Decision | Choice | Rationale |
|---|----------|--------|-----------|
| 1 | Source acquisition | GitHub-only, shallow clones, managed in `.gitignore` | Clean state management. `update` = delete + re-clone. No submodules. |
| 2 | MCP tool routing | Explicit `package` parameter on every tool | Stateless. Scales to N packages. MCP tools should not have session state. |
| 3 | Metadata format | JSON (`book.json`), not YAML | Pydantic serializes to JSON natively. Zero extra dependencies. |
| 4 | Interactive CLI | Future TODO. Bare command prints help. | Core functionality first. Interactive wizard is a UX layer on top. |
| 5 | Venv validation | Removed — `uv` handles isolation | `uv run --project` guarantees the correct environment. No manual check needed. |
| 6 | Index scoping | No scoping. Index everything. | Scoping is bad UX — users shouldn't need to know internal directory structure. |
| 7 | Weight persistence | `weights.json` separate from `01_index.json` | Rebuild does not destroy learned weights. Different files, different lifecycles, different triggers. |
| 8 | Context file caching | No cache. Re-parse every time (~1ms). | Caching saves 1ms but adds invalidation complexity. Not worth it. |
| 9 | Central config | `mcpmenago.json` — minimal, one file for global settings | Persistent `learn_dirs` override. Not coupled to any specific project. Pydantic model. |
| 10 | Naming convention | library/books metaphor. `book.json`, `weights.json` | Consistent, memorable. "book" = one managed repository. "library" = collection of books. |
| 11 | Directory name | `mcpmenago/` not `mcp/` | Standalone identity. Clear branding. Ready for extraction. |
| D1 | MCP server execution | `uv run --project ./mcpmenago` | MCP ecosystem standard. All official Anthropic servers use `uv`/`uvx`. Eliminates venv management. |
| D2 | Package structure | `mcpmenago/mcpmenago/` source layout | Proper Python package structure. `uv` can build and install correctly. Ready for PyPI. |
| D3 | Deps in pyproject.toml | Runtime + optional dev deps declared | `uv run --project` reads `pyproject.toml` natively. No separate requirements files needed. |
| D4 | Tests location | `mcpmenago/tests/` inside project | Self-contained. Copy-paste extraction to own repo works without moving tests. |
| D5 | Index file naming | `01_index.json` (numbered prefix) | Signals stage 1 of a build pipeline. Future stages get `02_`, `03_` prefixes. File naming IS the contract — intermediate artifacts are inspectable on disk. |
| D6 | Analysis script | `scripts/analyze_index.py` → `01_index_analysis.json` | Standalone, no side effects. Produces metrics JSON to inform architecture decisions without reading the full index manually. |
| D7 | Indexing backend | Transitioning from tree-sitter to `scip` | scip produces richer, language-server-quality symbol data. tree-sitter approach retired. |
