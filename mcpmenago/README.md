# mcpmenago

**MCP Manager** — a standalone CLI tool + central MCP server that manages source code indexes for any GitHub repository. Point it at a repo, it clones and indexes it, then exposes the symbols via a single MCP server. Built on FastMCP and tree-sitter.

Abbreviated from "MCP Manager" — easy to remember, easy to type.

## Installation

```bash
pip install -e .  # from mcpmenago/ directory
```

This registers the `mcpmenago` CLI alias.

## Quick Start

```bash
# Add a repository
mcpmenago add https://github.com/rdkit/rdkit --lang python --lang cpp --head-ref Release_2025_09_6

# Learn which symbols your project uses (updates weights)
mcpmenago learn rdkit

# List managed books
mcpmenago list

# Show details for a book
mcpmenago show rdkit
```

## CLI Commands

| Command | Purpose |
|---------|---------|
| `mcpmenago add <github-url> --lang <lang> [--lang <lang>] [--head-ref <tag>] [--book-name <name>]` | Clone repo, build index |
| `mcpmenago remove <book>` | Remove book and all its data from library |
| `mcpmenago list` | List all managed books |
| `mcpmenago show <book>` | Show book config, index stats, weight summary |
| `mcpmenago rebuild <book>` | Rebuild index for a book (Layer 1). Weights preserved. |
| `mcpmenago rebuild --all` | Rebuild all indexes |
| `mcpmenago update <book> --head-ref <tag\|branch>` | Update to a new git ref (clean re-clone + rebuild) |
| `mcpmenago learn <book>` | Scan project imports, update weights (Layer 2) |
| `mcpmenago uninstall` | Remove everything — library, config, gitignore section |

### `add` flow
1. Venv pre-check: is the active python in the project's `.venv/`?
2. Validate GitHub URL format
3. `git clone --depth 1 [--branch <head-ref>] <url>` into `mcpmenago/library/<book>/repo/`
4. tree-sitter parses ALL source files for declared languages → `index.json`
5. Write `book.json` (BookMeta)
6. Append `mcpmenago/library/` to `.gitignore` under managed section (idempotent)

### `update` flow (clean state management)
1. Delete `mcpmenago/library/<book>/repo/` entirely
2. Fresh shallow clone at new ref — no incremental git, no submodules
3. Rebuild index from scratch
4. Update `book.json` with new `head_ref` + `head_ref_resolved` (commit SHA)
5. `weights.json` is preserved — it's independent of the index

### `remove` flow
1. Delete `mcpmenago/library/<book>/` directory entirely
2. If library is now empty, clean up `.gitignore` managed section

### `learn` flow
1. Read `learn_dirs` from `mcpmenago/mcpmenago.json` (defaults: `["src", "tests", "notebooks"]`)
2. Scan all `.py` and `.ipynb` files in those directories for imports matching the book's package
3. Write `weights.json`: `{symbol_name: 0.8}` for every discovered symbol
4. Symbols not found in project imports remain at default weight (`NOT_USED = 0.3`)

### `uninstall` flow
1. Delete entire `mcpmenago/library/` directory
2. Clean `.gitignore` managed section

### Book name derivation
Extracted from GitHub URL: `https://github.com/rdkit/rdkit` -> `rdkit` (last path segment). Override with `--book-name` flag.

## Filesystem Layout

```
mcpmenago/
├── server.py              # Central FastMCP server — all MCP tools
├── cli.py                 # Click CLI entry point -> `mcpmenago` alias
├── index_builder.py       # BookIndexer — tree-sitter parsing (Layer 1)
├── learn.py               # Dependency scanner — weight updates (Layer 2)
├── models.py              # Pydantic schemas + weight constants
├── venv_check.py          # Venv validation pre-check
├── mcpmenago.json         # Central config (user-editable, minimal)
├── README.md              # This file
└── library/               # .gitignored — all cloned repos + indexes
    └── <book>/            # One directory per managed repository
        ├── repo/          # Shallow clone from GitHub
        ├── book.json      # BookMeta (auto-generated by CLI, never manually edited)
        ├── index.json     # Unified index: modules + symbols (Layer 1 output)
        └── weights.json   # Learned weights (Layer 2 output, independent of index)
```

### `.gitignore` management
The CLI automatically appends a managed section to the project's `.gitignore`:
```gitignore
# [mcpmenago] automatically generated content — do not edit
mcpmenago/library/
```
All cloned repos and indexes are ignored. The tool code itself (`server.py`, `cli.py`, etc.) is committed.

## Central Config (`mcpmenago.json`)

Minimal, user-editable. Single source of truth for global settings.

```json
{
  "learn_dirs": ["src", "tests", "notebooks"],
  "supported_languages": ["python", "cpp"]
}
```

| Field | Purpose | Default |
|-------|---------|---------|
| `learn_dirs` | Directories to scan when running `mcpmenago learn`. Relative to project root. | `["src", "tests", "notebooks"]` |
| `supported_languages` | Languages with tree-sitter support available. Drives `--lang` validation. | `["python", "cpp"]` |

This file is NOT coupled to any specific project. Any project can onboard by setting its own `learn_dirs`.

## Schemas

### BookMeta (`book.json`)
Auto-generated by the CLI. Never manually edited.

```python
class BookMeta(BaseModel):
    name: str                              # "rdkit"
    github_url: str                        # "https://github.com/rdkit/rdkit"
    languages: list[str]                   # ["python", "cpp"]
    python_path: Path                      # .venv/.../site-packages/rdkit/
    head_ref: str | None = None            # "Release_2025_09_6"
    head_ref_resolved: str | None = None   # Actual commit SHA after clone
    index_built_at: str | None = None      # ISO timestamp of last index build
```

### BookIndex (`index.json`)
Unified index combining module metadata and symbol locations.

```python
class SymbolEntry(BaseModel):
    name: str              # "GetSubstructMatches"
    kind: str              # "function" | "class" | "method" | "template"
    file: str              # Relative path within repo
    start_line: int
    end_line: int
    signature: str         # First line / declaration

class ModuleSource(BaseModel):
    path: str              # "Chem/rdmolops.py" — relative to book source root
    lang: str              # "python", "cpp" — from supported_languages

class ModuleEntry(BaseModel):
    name: str                       # "Chem.rdmolops"
    sources: list[ModuleSource]     # Can span 1, 2, or N languages
    description: str
    key_functions: list[str]

class BookIndex(BaseModel):
    version: str
    modules: list[ModuleEntry]
    symbols: dict[str, list[SymbolEntry]]  # name -> entries (supports overloads)
```

### Weight Constants

```python
NOT_USED = 0.3           # Symbol exists in index, not found in user's project
DISCOVERED = 0.8         # Found in user's project imports via `learn`
# FREQUENTLY_USED = 1.0  # Future — requires usage tracking (see TODO section)
```

### McpMenagoConfig (`mcpmenago.json`)
```python
class McpMenagoConfig(BaseModel):
    learn_dirs: list[str] = ["src", "tests", "notebooks"]
    supported_languages: list[str] = ["python", "cpp"]
```

## MCP Server Tools

Central FastMCP server (`server.py`) replaces the previous `mcp/rdkit_mcp.py`. Every tool takes an explicit `package` parameter to route to the right book. STDIO transport.

| Tool | Purpose | Layers Used |
|------|---------|-------------|
| `get_symbol(name, package, context_file?)` | Exact symbol lookup — returns function/class body | L1 + L2 + L3 |
| `search_code(query, package, source?, context_file?)` | Tiered ripgrep search | L1 + L2 + L3 |
| `read_file(path, package, source?)` | Read one source file | L1 |
| `inspect_module(module_name, package)` | C++ signatures from compiled .so via inspect | L1 |
| `get_module_index(package, min_relevance?)` | Module map with weights | L1 + L2 |
| `list_directory(path, package)` | Browse source tree, names only | L1 |

### `.mcp.json` integration
After setup, the project's `.mcp.json` points to the central server:
```json
{
  "mcpServers": {
    "mcpmenago": {
      "command": ".venv/bin/python",
      "args": ["mcpmenago/server.py"]
    }
  }
}
```

## Three-Layer Ranking Architecture

The ranking system determines which results are returned and in what order. Three independent layers, each with its own persistence and lifecycle.

### Layer 1: Structural Index (tree-sitter)

**Built by:** `mcpmenago add` / `mcpmenago rebuild`
**Output:** `index.json` — every symbol, every file, all declared languages
**Persistence:** Survives until explicit rebuild or remove

Exhaustive parse of the cloned repository using tree-sitter. Per-language AST queries extract symbols (functions, classes, methods, templates) into a unified `SymbolEntry` schema. The consumer never sees language differences — the output contract is the same.

**Per-language adapters:**
- **Python** (`tree-sitter-python`): `function_definition`, `class_definition`
- **C++** (`tree-sitter-cpp`): `function_definition`, `template_declaration`, `class_specifier`
- Future languages: add a new tree-sitter grammar + query set. `supported_languages` in config drives validation.

No filtering. No scoping. Parse everything for the declared languages. The full index is the foundation that all other layers build on.

### Layer 2: Static Weights (dependency scanning)

**Built by:** `mcpmenago learn`
**Output:** `weights.json` — `{symbol_name: weight}` for discovered symbols
**Persistence:** Independent of `index.json`. Survives index rebuild.

Scans the user's project directories (from `learn_dirs` in `mcpmenago.json`) for import statements matching the book's package. Updates weights:
- Symbol found in project imports → `DISCOVERED = 0.8`
- Symbol not found → remains at `NOT_USED = 0.3` (default, not stored in file)

Manual and explicit. User calls `mcpmenago learn <book>` when their imports change. No automatic watching or caching. Re-parse is always fresh (~1ms, no correctness concerns).

**Why weights.json is separate from index.json:**
- `rebuild` (Layer 1) does not destroy learned weights
- `learn` (Layer 2) does not touch the structural index
- Layers are truly independent — different lifecycles, different triggers

### Layer 3: Query Context (per-request)

**Built by:** `context_file` parameter on each MCP tool call
**Output:** In-memory filtering/boosting of results
**Persistence:** None — computed fresh every request

At query time, the MCP server parses the `context_file` (the file the user is currently editing) for import statements. Results matching those imports are filtered/boosted to the top.

- Always fresh parse. No caching. ~1ms per parse. Zero invalidation bugs.
- If no `context_file` provided, falls back to Layer 2 weights for ordering.
- If no `learn` has been run, falls back to Layer 1 with equal weights.

### Token efficiency by layer combination

| Layers active | Example: `get_symbol("GetSubstructMatches")` | Approx tokens |
|---------------|-----------------------------------------------|---------------|
| L1 only | All 5 overloads, no ordering | ~2000 |
| L1 + L2 | All 5, sorted by weight, high-relevance first | ~800 (with limit) |
| L1 + L2 + L3 | Filtered to context imports, 1-2 hits | ~300 |

## Venv Pre-Check

Basic validation that runs automatically before `add` and `rebuild` commands. Ensures the active Python is in the project's `.venv/`.

**Current scope (basic):**
- Check if `sys.executable` is under the project root's `.venv/`
- If not: clear error message + proposed fix command
- Fail fast before doing any work

**See TODO section for the full interactive venv check planned for the future.**

## Design Principles

- **Flat is better than nested** — minimal directory depth, simple organization
- **Simple architecture is better than complex** — one server, one config, one index per book
- **Standalone and portable** — fully contained in `mcpmenago/`, no coupling to any specific project
- **User-centered UX** — minimal effort, good error messages, foolproof defaults
- **Layers are independent** — different files, different lifecycles, different triggers
- **Pydantic everywhere** — schemas, config, serialization. MongoDB-ready by design.

**Trigger:** User runs bare `mcpmenago` command with no subcommand

A deterministic state machine that walks the user through adding/managing books interactively. The concept is user-centered: zero effort, optimize retrieving relevant information.

**Key design principles:**

**Current state:** Bare `mcpmenago` prints help text. All operations use explicit CLI flags.

## Decisions Log

Key architectural decisions made during design, with rationale.

| # | Decision | Choice | Rationale |
|---|----------|--------|-----------|
| 1 | Source acquisition | GitHub-only, shallow clones, managed in `.gitignore` | Clean state management. `update` = delete + re-clone. No submodules. |
| 2 | MCP tool routing | Explicit `package` parameter on every tool | Stateless. Scales to N packages. MCP tools should not have session state. |
| 3 | Metadata format | JSON (`book.json`), not YAML | Pydantic serializes to JSON natively. Zero extra dependencies. Consistent with `index.json`. |
| 4 | Interactive CLI | Future TODO. Bare command prints help. | Core functionality first. Interactive wizard is a UX layer on top. |
| 5 | Venv validation | Basic pre-check now, full interactive check later | Basic check catches 90% of issues. Full check is an interactivity feature. |
| 6 | Index scoping | No scoping. Index everything. | tree-sitter is fast (~3-5s for 1000 files). Scoping is bad UX — users shouldn't know internal dir structure. |
| 7 | Relevance system | Three layers: structural index, static weights, query context | Layers are independent (separate files, lifecycles). Weight system is simple: binary discovered/not-discovered. |
| 8 | Weight persistence | `weights.json` separate from `index.json` | Rebuild (L1) doesn't destroy learned weights (L2). Layers are truly independent. |
| 9 | Context file caching | No cache. Re-parse every time (~1ms). | Caching saves 1ms but adds invalidation complexity. Not worth it. |
| 10 | Central config | `mcpmenago.json` — minimal, one file for global settings | Persistent `learn_dirs` override. Not coupled to any specific project. Pydantic model. |
| 11 | Naming convention | library/books metaphor. `book.json`, `index.json`, `weights.json` | Consistent, memorable. "book" = one managed repository. "library" = collection of books. |
| 12 | Directory name | `mcpmenago/` not `mcp/` | Standalone identity. Clear branding. Ready for extraction. |
